## Q1. Claude를 제일 많이 쓰는 용도는 무엇인가?

빈도순 5가지. 한 문장으로 요약하면 **"계획 문서 기반으로 payment-service 기능을 구현시키고, git 절차를 통째로 위임하는 개발 파트너"**.

1. **기능 구현·수정** (구현/수정 관련 362개) — "아래 요구사항 분석하여 구현해줘" + 명세 붙여넣기, 또는 `@.claude/plans/...` 계획 문서 참조 후 실행 지시.
2. **git 절차 대행** (관련 언급 666개) — 커밋·브랜치·이슈·PR 생성을 전부 위임. `/create-branch`(35) → 구현 → "커밋해줘"(118) → `/create-pr`(29) 파이프라인.
3. **계획 수립·브레인스토밍** (306개) — 코드 전에 계획 문서부터. `ExitPlanMode` 118회, `AskUserQuestion` 211회가 증거.
4. **디버깅** (305개) — 에러 로그/서버 로그 통붙여넣기 후 "문제 파악해봐".
5. **학습·이해** (질문/설명 202개 + `/grill-me` 27회) — 구현한 코드를 되물어 학습하는 용도. 다른 사용자에게 드문 특징적 패턴.

## Q2. 매번 반복하는 작업은 무엇인가?

| 반복 작업 | 빈도 | 현재 방식 |
|---|---|---|
| diff 파악 후 컨벤션 맞춰 커밋 | ~118회 | **자연어** ("커밋해줘.") |
| 이슈 → 브랜치 → PR 생성 | 각 25~35회 | 커스텀 스킬 (자동화 완료) |
| 계획 문서 참조 후 구현 재개 | `@.claude/plans/` 106회 | 자연어 + @참조 |
| 에러 로그 붙여넣고 원인 분석 | 수십 회 | 자연어 ("공유할게. 파악해봐.") |
| 테스트 실행 후 결과 보고 | 8회+ (암묵 다수) | 자연어 |
| 작업 후 문서 정리/갱신 | 19회 | 자연어 |
| 코드 되물어 학습 | 27회 | `/grill-me` 스킬 (자동화 완료) |

## Q3. 습관처럼 매번 다시 쓰는 지시문은 무엇인가?

패턴 추출 결과 (1,121개 중):

- **"커밋해줘." / "변경 사항 커밋해줘."** — 118회. 최다 반복 문장.
- **"@.claude/plans/{문서} ... 진행해줘 / 이어서 진행"** — 106회. 계획 문서 기반 작업 개시 관용구.
- **"아래 요구사항 분석하여(파악하여) 구현해줘/계획 세워줘."** — 34회. 명세 붙여넣기 서두.
- **"~ 에러 발생했어. 로그 공유할게. 문제 파악해봐."** — 디버깅 개시 관용구.
- **"왜 ~야?" / "설명해줘" / "너 생각은 어때?"** — 56회. 학습·검증 질문.
- **"이어서 진행해." / "진행해줘." / "좋아."** — 승인·재개 한 마디.

## Q4. 어떤 걸 재사용 가능한 skill로 만들면 좋은가?

ROI 순위:

1. **`/commit` (커밋 스킬) — 1순위.** 118회 반복인데 `/create-commit`은 1회만 사용. diff 분석 → `type: 한국어 설명` + 본문 불릿 + AI 트레일러까지 git-conventions.md 규칙을 스킬에 내장하면 매번 규칙 파일을 다시 읽는 비용도 사라짐. "커밋하고 푸시" 변형도 옵션(`--push`)으로 흡수.
2. **`/debug-log` (로그 분석 스킬).** 에러 로그 붙여넣기 세션이 많고 최대 31만 자 통붙여넣기 사례 존재. 스킬이 "로그를 scratchpad 파일로 저장 → 결정적인 라인만 추출 → systematic-debugging 진입" 절차를 강제하면 컨텍스트 낭비와 분석 품질 둘 다 개선.
3. **`/resume-plan` (계획 재개 스킬).** `@.claude/plans/` 참조 106회. "계획 문서 읽기 → 완료/미완료 Task 판별 → 다음 Task부터 실행"을 정형화. 지금은 매번 문맥을 자연어로 다시 설명함.
4. **트러블슈팅 문서 자동화 보강.** 규칙(`troubleshooting-doc-format.md`)은 있는데 산출물 0개 — 규칙만으론 실행이 안 됨. 디버깅 세션 종료 시점에 트리거되는 스킬(또는 Stop 훅)로 승격 필요.
5. 이슈/브랜치/PR/grill-me는 **이미 스킬화 완료** — 추가 작업 불필요. 잘 되고 있는 부분.

## Q5. 모델이 무엇을 미리 알고 있으면 Fable 5 경험의 90%를 낼 수 있는가?

전제 사실: 지난 30일 메시지의 **83%가 이미 Sonnet**(4-6/5)이었고 Fable은 3% 미만. 즉 "Fable 경험"의 대부분은 모델 지능이 아니라 **미리 코드화해둔 컨텍스트**가 만들고 있었음. 그 컨텍스트의 목록:

**이미 코드화되어 있는 것 (유지가 곧 90%의 기반):**
- `CLAUDE.md` + `.claude/rules/*` — 클린 아키텍처 패키지 계약, git 컨벤션, Flyway 규칙, 테스트 정책(Testcontainers), 언어 정책. 모델이 바뀌어도 산출물 품질을 지탱하는 핵심.
- `.claude/docs/*` — API 설계·DB 스키마·Kafka 이벤트 계약. 이게 없으면 어떤 모델이든 이벤트 계약을 추측으로 구현함.
- `.claude/plans/*` — 작업의 "왜"가 담긴 결정 기록. 세션을 `/clear`로 잘게 끊는 사용 습관(304회)에서 세션 간 연속성을 담당하는 실질적 메모리.
- 커스텀 스킬(create-issue/branch/pr, grill-me) — 절차 지식의 외장화.

**아직 사람 머릿속에만 있어서 매번 다시 설명하는 것 (추가로 적어두면 격차 해소):**
1. **디버깅 이력** — 같은 계열 에러(Placeholder 미해석, CI 실패, Kafka 테스트 충돌 등)를 반복 조사. 트러블슈팅 문서가 0개라 매번 처음부터. Q4-4가 해결책.
2. **검증 명령 모음** — "이 변경엔 이 테스트, 이 curl, 이 로그 확인"이라는 암묵지. `verify` 계열 프로젝트 스킬로 명문화하면 구현 후 검증 품질이 모델 무관하게 유지됨.
3. **도메인 상태 머신의 "왜"** — 결제/환불 상태 전이의 비즈니스 근거. 계획 문서에 산재. `docs/` 한 곳에 정리하면 새 세션이 추측하지 않음.
4. **자주 하는 선택의 기본값** — AskUserQuestion 211회 중 상당수가 반복 선택(테스트 범위, 문서 위치, 커밋 분리 여부). 선호를 CLAUDE.md나 메모리에 박으면 질문 턴 자체가 감소.

**남는 10%** — 큰 설계 판단(환불 흐름 재설계 같은), 미묘한 트레이드오프 리뷰, 긴 디버깅 체인의 가설 수립. 이건 컨텍스트로 대체 안 되는 모델 지능 영역이므로, "평소 Sonnet + 설계·리뷰 순간만 상위 모델" 스위칭(`/model` 49회 — 이미 하고 있는 습관)이 최적.
