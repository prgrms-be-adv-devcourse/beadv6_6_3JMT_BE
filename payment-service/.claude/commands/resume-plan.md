사용자가 입력한 내용: **$ARGUMENTS**

아래 절차를 단계별로 수행하라.

## 1단계: 태스크 문서 결정

- `$ARGUMENTS`에 문서명(또는 일부)이 있으면 `.claude/plans/`에서 해당 파일을 찾는다. 부분 일치 허용(예: `398` → `398-refund-flow-redesign-tasks.md`).
- 생략 시 `.claude/plans/` 직하(archive 제외)에서 가장 최근 수정된 `*-tasks.md`를 선택한다.
- `-tasks.md`가 없고 계획 문서(`{번호}-{slug}.md`)만 있으면 그 문서를 사용하되, "태스크 문서가 아니라 계획 문서입니다 — 구현 순서 섹션 기준으로 진행합니다."라고 알린다.
- 대응하는 계획 문서(`-tasks` 없는 동명 파일)가 있으면 **함께 읽는다** — 설계 결정의 "왜"가 거기에 있다.

## 2단계: 완료/미완료 Task 판별

태스크 문서의 `### Task N: {제목}` 단위로 완료 여부를 판별한다. **체크박스만 믿지 않는다** — 체크박스 갱신이 누락된 사례가 실제로 있다(398 문서: 전체 미체크 상태로 머지 완료). 아래 순서로 교차 확인한다.

```bash
git log --oneline -30
git log --oneline origin/develop -15
```

1. **체크박스**: Task 내 모든 Step이 `- [x]`면 완료 후보.
2. **git log 교차 확인**: 각 Task의 마지막 Step은 `Commit`이므로, Task 내용과 대응하는 커밋이 현재 브랜치(또는 develop)에 존재하는지 커밋 메시지로 확인한다.
3. **판별 충돌 시**: git log에 커밋이 있으면 완료로 간주한다(체크박스 미갱신). 반대로 체크박스만 체크되고 커밋이 없으면 미완료로 간주하고 그 사실을 보고한다.
4. 필요하면 Task가 만들었을 파일의 존재를 직접 확인해 보강한다.

## 3단계: 재개 지점 확인

판별 결과를 아래 형식으로 보여주고 사용자 확인을 받는다.

```
=== 계획 재개 ===
문서: 398-refund-flow-redesign-tasks.md
브랜치: <현재 브랜치>

Task 1: <제목> — ✅ 완료 (커밋 abc1234)
Task 2: <제목> — ✅ 완료 (커밋 def5678, 체크박스 미갱신)
Task 3: <제목> — ⬜ 미완료

다음 작업: Task 3 — <제목>
================

Task 3부터 진행할까요?
```

- 모든 Task가 완료면 "이 계획은 모든 Task가 완료된 상태입니다."라고 보고하고 종료한다.
- 사용자가 다른 Task를 지정하면 그 Task부터 진행한다.

## 4단계: Task 실행

확인받은 Task부터 문서의 Step 순서대로 실행한다.

- **Task = 1 커밋** (`plan-doc-format.md`) — Task 내부를 여러 커밋으로 쪼개지 않고, 마지막 Step에서 `/create-commit` 절차로 커밋한다.
- Step을 완료할 때마다 태스크 문서의 해당 체크박스를 `- [x]`로 갱신한다(2단계 문제의 재발 방지). **단, 체크박스 갱신만으로 별도 커밋을 만들지 않는다** — 갱신 내용은 그대로 두었다가 해당 Task의 마지막 Step(`Commit`)에서 코드 변경분과 함께 한 커밋으로 묶는다.
- 문서의 Global Constraints 섹션이 있으면 그 제약을 준수한다.
- 한 Task가 끝나면 다음 Task로 자동 진행하지 않고, 완료 보고 후 사용자 지시를 기다린다.
