사용자가 입력한 내용: **$ARGUMENTS**

아래 절차를 단계별로 수행하라. 커밋 형식의 원본 규칙은 `.claude/rules/git-conventions.md`다 — 이 문서와 충돌하면 규칙 문서가 우선한다.

## 1단계: 인자 파싱

- `$ARGUMENTS`에 `--push`가 있으면 7단계에서 푸시까지 수행한다.
- 나머지 텍스트는 커밋 메시지 작성 힌트로 사용한다(없어도 됨 — diff에서 직접 파악).

## 2단계: 변경사항 수집

```bash
git status --short
git diff --stat
git diff
git log --oneline -5
```

- 변경이 하나도 없으면(스테이징 포함) "커밋할 변경사항이 없습니다."라고 안내하고 중단한다.
- 현재 브랜치가 `develop` 또는 `main`이면 **즉시 중단**하고 "보호 브랜치입니다. 작업 브랜치를 먼저 만들어 주세요. (/create-branch)"라고 경고한다.
- 최근 커밋 로그는 이 저장소의 실제 메시지 스타일 참고용이다.

## 3단계: 커밋 단위 판단

변경 파일들이 **하나의 논리적 변경**인지 확인한다.

- 성격이 다른 변경이 섞여 있으면(예: 기능 구현 + 무관한 설정 변경) 분리 커밋을 제안하고 사용자에게 확인한다.
- 하나의 변경이면 전체를 스테이징 대상으로 한다.

## 4단계: 커밋 메시지 작성

`git-conventions.md` 형식을 따른다.

- **제목**: `type: 한국어 설명` — type은 `feat`/`fix`/`docs`/`chore`/`test`/`style`/`refactor` 중 diff 성격에 맞는 것. scope 미사용.
- **본문**: 변경 폭이 크거나 맥락 설명이 필요할 때만 제목 뒤 빈 줄 후 불릿(`-`)으로 작성. 작은 변경은 제목만.
- **`#이슈번호`를 본문에 쓰지 않는다** — 이슈 추적은 브랜치명과 PR의 몫.
- **AI 트레일러**: AI가 작업에 기여한 커밋이면 본문 마지막에 빈 줄 후 추가. 모델명은 실제 작업에 사용한 버전으로 기입한다(버전 생략 금지).

  ```
  Co-Authored-By: Claude {모델명 버전} <noreply@anthropic.com>
  ```

## 5단계: 초안 제시 및 동의 획득

아래 형식으로 초안을 보여준다.

```
=== 커밋 초안 ===
브랜치: <현재 브랜치>
스테이징 대상:
(git status --short 결과)

메시지:
(작성된 커밋 메시지 전체)
================

이 내용으로 커밋할까요? (수정이 필요하면 말씀해 주세요)
```

수정 요청이 있으면 반영 후 다시 보여준다. **명시적 동의를 받기 전까지는 git add/commit을 실행하지 않는다.** 단, 사용자가 이 커맨드를 호출하며 이미 커밋을 지시한 맥락이 명확하고 변경 내용이 단순하면(파일 1~2개, 단일 성격) 초안 제시와 동시에 진행해도 된다.

## 6단계: 커밋 실행

```bash
git add <대상 파일들>
git commit -m "<메시지>"
git log -1 --stat
```

- `git add -A`보다 파악한 파일을 명시적으로 지정하는 것을 우선한다(의도치 않은 파일 포함 방지).
- 실행 후 커밋 해시와 변경 통계를 보고한다.

## 7단계: 푸시 (--push 인자가 있을 때만)

```bash
git push -u origin <현재 브랜치>
```

`--push`가 없으면 푸시하지 않는다. 사용자가 대화에서 "커밋하고 푸시해"라고 요청한 경우는 `--push`와 동일하게 취급한다.
