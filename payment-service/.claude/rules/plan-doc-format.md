payment-service `.claude/plans/` 문서 작성 형식 규칙. 새 계획/구현 문서 작성 시 따른다. 기존 `archive/` 파일에는 소급 적용하지 않는다.

## 배경

`.claude/plans/`의 문서는 두 종류로 나뉜다 — 작업 착수 전 **계획 문서**(`{NN}-{slug}.md`)와 작업 완료 후 **구현 문서**(`{NN}-{slug}-impl.md`). 목적이 다르므로 템플릿도 분리한다.

이 문서들은 훗날 포트폴리오·이력서 작성 시 본인이 직접 발췌하는 용도로도 쓰인다. 발췌 시 중요한 것은 기술적 의사결정 근거, 문제해결 서사, 정량적 성과 — 이를 위해 구현 문서는 맨 위에 3줄 요약을 둔다. 본문을 이력서 톤으로 미리 각색하지는 않는다.

## 1. 계획 문서 템플릿 — 뼈대만 고정, 본문은 자유

```markdown
# {작업명} 구현 계획

{한 줄 요약: 무엇을 / 왜}

---

## 배경 및 목표

...

## (본문 — 작업 성격에 맞는 섹션 자유 구성: 확정 사항 / 설계 결정 / 구현 순서 / 트레이드오프 / 테스트 케이스 등)

...
```

고정: 제목, 한 줄 요약, `---` 1개(첫 섹션 앞). 본문 섹션 이름·개수는 작업에 맞게 자유 — 억지로 통일하지 않는다.

## 2. 구현 문서 템플릿 — 공통 섹션까지 고정

```markdown
# {작업명} 구현 결과

> 계획 문서: `{NN}-{slug}.md`
> 브랜치: `{브랜치명}` (있으면) / 작성일: {YYYY-MM-DD} (있으면)

---

## 요약
- 문제: {왜 착수했는지 — 기존 문제/제약}
- 해결: {기술적으로 어떻게 풀었는지 — 핵심 의사결정}
- 성과: {정량적 결과 — %, 시간, 건수 등. 숫자 없으면 정성적 효과라도}

## 커밋 이력

| 커밋 | 내용 |
|---|---|

## 계획 대비 변경 결정

계획과 실제 구현이 달라진 부분과 이유.

## 트러블슈팅

구현 중 발견/해결한 문제.

## 테스트 결과

## 잔여 작업

(없으면 섹션 자체 생략)
```

고정: 제목은 `{작업명} 구현 결과` 접미사 패턴, 메타데이터는 제목 바로 아래 blockquote, `---` 1개. **`## 요약`은 필수**(3줄, 이력서 발췌용) — 문제/해결/성과 순서 고정. 그 아래 4개 섹션(커밋 이력/계획 대비 변경 결정/트러블슈팅/테스트 결과/잔여 작업)은 **내용 있을 때만** 이 이름·순서로 사용한다.
