payment-service `.claude/plans/` 문서 작성 형식 규칙. 새 계획 문서 작성 시 따른다. 기존 `archive/` 파일에는 소급 적용하지 않는다.

## 배경

`.claude/plans/`에는 작업 착수 전 **계획 문서**(`{NN}-{slug}.md`)를 작성한다.

이 문서는 훗날 포트폴리오·이력서 작성 시 본인이 직접 발췌하는 용도로도 쓰인다. 발췌 시 중요한 것은 기술적 의사결정 근거, 문제해결 서사, 정량적 성과.

## 계획 문서 템플릿 — 뼈대만 고정, 본문은 자유

```markdown
# {작업명} 구현 계획

{한 줄 요약: 무엇을 / 왜}

---

## 배경 및 목표

...

## (본문 — 작업 성격에 맞는 섹션 자유 구성: 확정 사항 / 설계 결정 / 구현 순서 / 트레이드오프 / 테스트 케이스 등)

...
```

고정: 제목, 한 줄 요약, `---` 1개(첫 섹션 앞). 본문 섹션 이름·개수는 작업에 맞게 자유 — 억지로 통일하지 않는다.

## 브레인스토밍 결론 섹션 (확정 사항 / 설계 결정 등)

"확정 사항", "설계 결정" 등 브레인스토밍 결론을 담는 섹션은 항목을 한 줄 요약으로 나열하지 않는다. 각 항목마다 아래 두 가지를 문장으로 서술한다.

- **무엇을 정했는지**
- **왜 그렇게 정했는지** — 배경, 경쟁하는 대안 대비 근거, 결정을 촉발한 사실

## 태스크 목록 문서 (writing-plans 산출물)

파일명 규칙은 [superpowers-workflow.md](superpowers-workflow.md) 참조.

각 Task는 **하나의 커밋 단위**다 — Task의 마지막 Step은 항상 `Step: Commit`이며, Task 내부를 여러 커밋으로 쪼개지 않는다.
