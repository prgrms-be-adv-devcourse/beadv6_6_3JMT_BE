# GitHub 이슈 생성

레포의 이슈 템플릿을 단일 진실 공급원으로 삼아, 사용자가 설명한 내용을 템플릿 형식에 맞게
채우고 `gh issue create`로 실제 이슈를 등록한다.

핵심 원칙은 두 가지다. 첫째, 템플릿 내용을 이 스킬에 복사해두지 않고 **실행 시점에
`.github/ISSUE_TEMPLATE/`에서 직접 읽는다**. 템플릿이 바뀌어도 스킬을 고칠 필요가 없도록
하기 위해서다. 둘째, 이슈 본문은 **사람이 직접 쓴 것처럼 담백하게** 채운다. 자세한 톤 규칙은
아래 "톤 가이드"를 따른다.

## 워크플로우

### 1. 템플릿 종류 결정 (버그 vs 기능)

- 사용자가 명시하면("버그 이슈", "기능 요청") 그대로 따른다.
- 명시가 없으면 설명 내용으로 판단한다. 무언가 동작하지 않거나 에러·장애를 말하면 버그,
  새로 만들거나 바꾸고 싶은 걸 말하면 기능이다.
- 판단이 애매하면 추측해서 진행하지 말고 사용자에게 한 번 확인한다. 잘못된 템플릿으로
  이슈를 만들면 되돌리는 비용이 크기 때문이다.

### 2. 템플릿 읽기

`.github/ISSUE_TEMPLATE/` 디렉토리에서 해당 파일을 읽는다.
- 버그: `bug_report.md`
- 기능: `feature_request.md`

파일명이 다를 수 있으니 디렉토리를 먼저 확인한다. 읽은 뒤 frontmatter(`---`로 감싼 부분)와
본문을 분리한다.

**frontmatter 처리 (중요)**: 이 frontmatter는 GitHub 웹 UI에서 템플릿을 고를 때만 자동
해석된다. `gh issue create` CLI는 frontmatter를 해석하지 않으므로, 이 스킬이 직접 파싱해서
CLI 플래그로 변환해야 한다.

- `title`: 접두사(`[BUG]`, `[FEATURE]` 등)만 사용한다. 그 뒤에 붙은
  "간단한 버그 설명을 작성하세요." 같은 문구는 **플레이스홀더이므로 실제 제목으로 교체**한다.
- `labels`: 값을 추출해 `--label`로 넘긴다 (예: `bug`, `feature`).
- `assignees`: 비어 있으면 아무도 할당하지 않는다.

### 3. 본문 채우기

본문의 각 섹션을, 사용자가 준 정보로 채운다. 템플릿에 들어 있는 안내 문구
(예: "어떤 버그가 발생했는지 명확하고 자세하게 설명해주세요.")는 그대로 두지 말고 **실제
내용으로 대체**한다. 섹션 제목과 구조(이모지 포함)는 템플릿 그대로 유지한다.

### 4. 필수 항목 보강

정보가 부족하면, 필요한 것을 **한 번에 모아** 사용자에게 되묻는다. 매번 한 항목씩 따로
물어보면 번거롭기 때문이다. 선택 항목이 비는 건 정상이니 되묻지 않는다.

필수로 채워야 하는 항목:
- **버그**: 버그 설명 / 재현 단계 / 예상 결과 / 실제 결과
- **기능**: 문제 제기 / 제안하는 기능

선택 항목(되묻지 않음): 환경, 스크린샷, 대안, 추가 정보 등. 사용자가 정보를 줬으면 채우고,
없으면 비워두거나 섹션을 짧게 남긴다.

### 5. 이슈 생성

`gh`로 이슈를 만든다. 본문은 임시 파일이나 `--body-file`을 쓰면 줄바꿈·마크다운이 깨지지
않아 안전하다.

```bash
gh issue create \
  --title "[BUG] 결제 완료 후 정산 금액이 0원으로 표시됨" \
  --label bug \
  --body-file /tmp/issue-body.md
```

`--body`에는 **frontmatter를 뺀 본문 섹션만** 넣는다. frontmatter가 본문에 그대로 들어가면
안 된다.

생성이 끝나면 `gh`가 출력한 이슈 URL을 사용자에게 그대로 전달한다.

## 톤 가이드 (본문 작성 규칙)

이슈 본문은 동료가 슬랙이나 이슈 트래커에 직접 적은 것처럼 담백해야 한다. AI가 쓴 티가 나면
팀원들이 읽을 때 어색하고 신뢰가 떨어진다. 그래서:

- 과장된 수식어("획기적인", "강력한", "완벽한")를 쓰지 않는다.
- 불필요한 친절체나 설명 늘리기를 하지 않는다. 사실 위주로 짧게 쓴다.
- 이모지를 본문 내용에 새로 추가하지 않는다 (템플릿 섹션 제목의 이모지는 유지).
- 모든 문장을 굵게 강조하거나, 과한 머리말을 붙이는 정형화된 구조를 피한다.
- 핵심만 적는다. 한 문장으로 될 걸 세 문장으로 늘리지 않는다.

예시로 감을 잡자.

**나쁜 예 (AI 티)**:
> 사용자께서 결제를 완료하신 후에 정산 금액이 안타깝게도 0원으로 표시되는 매우 중요한
> 이슈가 발생하고 있습니다! 이는 반드시 신속하게 해결되어야 할 핵심적인 문제입니다.

**좋은 예 (담백)**:
> 결제 완료 후 정산 금액이 0원으로 표시됨. 실제 정산 데이터는 정상인데 화면 표시만 0원.

## 엣지 케이스

- **`gh` 미설치 / 미인증**: `gh auth status`로 확인되지 않으면 이슈를 만들지 말고, 설치
  (`brew install gh`)나 인증(`gh auth login`)이 필요하다고 안내한다. 사용자가 직접
  실행해야 하므로 `! gh auth login`처럼 세션에서 실행하도록 제안해도 좋다.
- **템플릿 디렉토리/파일 없음**: `.github/ISSUE_TEMPLATE/`나 해당 템플릿 파일이 없으면
  추측으로 본문을 지어내지 말고, 사용자에게 알리고 멈춘다.
- **여러 라벨**: `labels`에 값이 여러 개면 `--label a --label b`로 모두 넘긴다.

## 다른 스킬과의 경계

이 스킬은 **이슈** 생성 전용이다. PR(Pull Request) 생성은 PR 전용 스킬에서 다루므로, 사용자가
PR을 만들려 하면 이 스킬을 쓰지 않는다.
