# GitHub PR 생성

레포의 PR 템플릿을 단일 진실 공급원으로 삼아, 현재 브랜치의 커밋·diff를 분석해 템플릿을 채우고,
사용자 승인을 받은 뒤 `gh pr create`로 Pull Request를 만든다.

핵심 원칙은 세 가지다. 첫째, 템플릿 내용을 이 스킬에 복사해두지 않고 **실행 시점에
`.github/ISSUE_TEMPLATE/pull_request_template.md`에서 직접 읽는다**. 특히 체크리스트는 앞으로
컨벤션·검증 rule이 정해지면 템플릿에서 바뀔 부분이므로, 스킬에 항목을 박아두면 안 된다. 템플릿이
바뀌면 스킬을 안 고쳐도 자동 반영돼야 한다. 둘째, PR 본문은 **사람이 직접 쓴 것처럼 담백하게**
채운다. 톤 규칙은 아래 "톤 가이드"를 따른다. 셋째, **승인 전에는 push도 PR 생성도 하지 않는다**.

## 워크플로우

### 1. 사전 점검

PR을 만들기 전에 막힐 조건부터 확인한다. 여기서 걸리면 본문을 만들 필요가 없으니 먼저 본다.

- `gh auth status`로 `gh` 설치·인증 확인. 안 되면 멈추고 안내한다 (엣지 케이스 참고).
- 현재 브랜치가 base 후보(`develop`, `main`) 자체면 PR을 만들 수 없다. 알리고 멈춘다.

### 2. 컨텍스트 수집 (git)

base 브랜치를 잠정적으로 정하고(아래 4단계에서 확정) 그에 대비한 변경을 모은다.

```bash
git rev-parse --abbrev-ref HEAD          # 현재(head) 브랜치
git log <base>..HEAD --format="%s%n%b"   # base 대비 커밋 메시지
git diff <base>...HEAD --stat            # 변경 파일 요약
git diff <base>...HEAD                   # 실제 diff
```

- base 대비 커밋이 하나도 없으면 만들 PR이 없다. 알리고 멈춘다.
- diff에서 테스트 파일(`src/test/...`, `*Test.java`, `*.test.*` 등) 변경 여부를 따로 본다.
  테스트 계획 섹션을 채울 근거로 쓴다.

### 2.5. 룰 검증 게이트 (verify-rules) — 통과 못 하면 중단

본문을 채우기 전에 **verify-rules 스킬을 호출**해 변경 코드가 프로젝트 룰 7종을 지키는지 검증한다.

- verify-rules가 **위반(FAIL)을 하나라도 보고하면 PR을 만들지 않는다.** 위반 목록(대상:위치·사유)을
  사용자에게 그대로 보여주고, 수정 후 다시 시도하도록 안내한 뒤 멈춘다. `git push`도 `gh pr create`도
  하지 않는다.
- 전 룰이 PASS/N/A면 게이트 통과다. verify-rules가 만든 체크리스트 결과(룰별 `[x]`)를 보관했다가
  4단계 체크리스트 섹션에 그대로 반영한다.

### 3. 템플릿 읽기

`.github/ISSUE_TEMPLATE/pull_request_template.md`를 읽는다. 파일이 없으면 멈춘다.

PR 템플릿에는 frontmatter가 없는 게 보통이다(`gh pr create`는 본문을 그대로 받는다). 섹션 구조와
**체크리스트 항목을 파일에서 그대로 가져온다.** 항목을 외워서 쓰거나 스킬에 적힌 예시로 대체하지
않는다 — 지금 템플릿에 무엇이 있는지가 유일한 기준이다.

### 4. 본문 채우기

템플릿의 각 섹션을 커밋·diff 근거로 채운다. 템플릿의 안내 문구("어떤 작업을 했는지 설명해주세요."
같은)는 그대로 두지 말고 실제 내용으로 대체한다. 섹션 제목·구조(이모지 포함)는 템플릿 그대로 둔다.

섹션별로:

- **설명 / 변경 사항 요약**: 커밋과 diff를 보고 무엇을, 왜 바꿨는지 적는다.
- **테스트 계획**: 테스트 파일 변경이 있으면 어떤 테스트를 추가·수정했는지 적고, 없으면 솔직히
  그 상태를 적는다(없는 테스트를 했다고 지어내지 않는다).
- **체크리스트**: 2.5단계 verify-rules가 통과시킨 결과를 그대로 반영한다. 게이트를 통과했으므로
  7개 룰 항목은 `[x]`로 체크한다. N/A였던 항목은 옆에 `(해당 변경 없음)`을 덧붙인다. 스킬이 임의로
  룰 위반을 다시 판단하지 않는다 — 판단은 verify-rules가 이미 했다.
- **관련 이슈**: 브랜치명·커밋 메시지에서 `#숫자`를 찾으면 `Closed #숫자` / `Related #숫자`로
  채운다. 못 찾으면 템플릿의 placeholder를 그대로 둔다.
- **설계 문서 / 리뷰 참고 / 추가 정보**: 자동으로 알 수 없는 항목이다. 사용자가 준 정보가 있으면
  채우고, 없으면 placeholder를 그대로 둔다. 추측으로 링크나 내용을 지어내지 않는다.

채운 본문은 임시 파일에 쓴다 (`/tmp/pr-body.md`). 줄바꿈·마크다운이 안 깨지고, 다음 단계에서
`--body-file`로 그대로 넘길 수 있다.

### 5. base 브랜치 확정 + 승인

생성 직전에, 사용자에게 다음을 한 번에 보여주고 승인을 받는다.

- **base ← head**: 현재 브랜치가 `feat/*`면 base로 `develop`을 제안한다. base는 추측만으로
  확정하지 말고 사용자에게 확인받는다.
- **PR 본문 초안**: 4단계에서 채운 전체 본문.

승인을 받기 전에는 `git push`도 `gh pr create`도 실행하지 않는다. 사용자가 base나 본문을
고치자고 하면 반영하고 다시 보여준다.

### 6. 생성

승인되면 원격에 브랜치를 올리고 PR을 만든다.

```bash
git push -u origin <head-branch>
gh pr create \
  --base <base> \
  --head <head-branch> \
  --title "<제목>" \
  --body-file /tmp/pr-body.md
```

- 제목은 대표 커밋이나 작업 내용을 커밋 컨벤션(`type: 설명`)에 맞춰 한 줄로 짓는다.
- 생성이 끝나면 `gh`가 출력한 PR URL을 사용자에게 그대로 전달한다.

## 톤 가이드 (본문 작성 규칙)

PR 본문은 동료가 직접 적은 것처럼 담백해야 한다. AI가 쓴 티가 나면 리뷰어가 읽을 때 어색하고
신뢰가 떨어진다. 그래서:

- 과장된 수식어("획기적인", "강력한", "완벽한")를 쓰지 않는다.
- 불필요한 친절체나 설명 늘리기를 하지 않는다. 사실 위주로 짧게 쓴다.
- 이모지를 본문 내용에 새로 추가하지 않는다 (템플릿 섹션 제목의 이모지는 유지).
- 모든 문장을 굵게 강조하거나 과한 머리말을 붙이는 정형화된 구조를 피한다.
- 핵심만 적는다. 한 문장이면 될 걸 세 문장으로 늘리지 않는다.

**나쁜 예 (AI 티)**:
> 이번 PR에서는 정산 도메인의 핵심 로직을 매우 견고하게 구현하였으며, 이를 통해 시스템의
> 안정성을 획기적으로 향상시켰습니다!

**좋은 예 (담백)**:
> 정산 금액 계산 로직 추가. 결제 건별로 수수료 차감 후 금액을 산출한다. 멱등성 위해 source_hash로
> 중복 처리 막음.

## 엣지 케이스

- **`gh` 미설치 / 미인증**: `gh auth status`로 확인되지 않으면 PR을 만들지 말고, 설치
  (`brew install gh`)나 인증(`gh auth login`)이 필요하다고 안내한다. 인증은 사용자가 직접
  해야 하므로 `! gh auth login`처럼 세션에서 실행하도록 제안해도 좋다.
- **현재 브랜치가 base(develop/main)**: PR을 만들 수 없다. 작업 브랜치로 옮기라고 안내하고 멈춘다.
- **base 대비 커밋 없음**: 올릴 변경이 없다. 알리고 멈춘다.
- **PR 템플릿 파일 없음**: 본문 구조를 지어내지 말고, 사용자에게 알리고 멈춘다.
- **이미 같은 head의 PR 존재**: `gh pr list --head <branch>`로 확인된다면 중복 생성하지 말고,
  기존 PR URL을 알려주고 멈춘다.
- **이슈 번호·설계 문서 못 찾음**: 정상이다. placeholder를 그대로 두고 진행한다. 사용자에게
  굳이 되묻지 않는다 — PR 생성 후 GitHub에서 직접 채워도 된다.

## 다른 스킬과의 경계

- **create-github-issue**: 이슈 생성 전용. 대상이 "이슈"면 그쪽, "PR"이면 이 스킬이다.
- **finishing-a-development-branch**: 브랜치를 어떻게 통합할지(merge / PR / cleanup) **결정**하는
  스킬이다. 이 스킬은 "PR로 가기로 이미 정한" 뒤 템플릿을 채워 실제로 생성하는 실행 단계다.
- **review / code-review / requesting-code-review**: diff를 **리뷰**하는 스킬이다. PR "생성"과는
  무관하다.
- **verify-rules**: PR 본문을 채우기 전에 이 스킬이 호출하는 룰 검증 게이트다. 통과하지 못하면
  PR을 만들지 않는다.
