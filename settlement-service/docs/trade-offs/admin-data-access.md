# 어드민 데이터 접근 — 직접 DB 커넥션 vs gRPC 조회

파이널에서 어드민 페이지 전용 모듈(admin-service)이 생기고, 정산 어드민 API 가 1호로
이관된다(`../architecture/admin-module-separation.md`). 그러면 어드민이 정산 데이터를
어떻게 읽느냐가 문제가 된다 — 정산 DB 에 **직접 커넥션**을 맺어 읽을 것인가, 정산
서비스에 **gRPC 로 물어** 받을 것인가.

강사 피드백은 전자였다. 어드민은 여러 도메인 데이터를 모아 보여주는 모듈이니, 각 서비스
DB 에 커넥션을 다 맺어놓고 필요한 걸 직접 읽으라는 취지다. 화면을 추가할 때마다 상대
서비스에 API 를 뚫지 않아도 되니 어드민 개발 속도가 빠르다.

> 서비스 간 동기 호출을 REST 로 할지 gRPC 로 할지는 `internal-sync-transport.md` 에서
> 이미 gRPC 로 정했다. 이 문서는 그 앞 단계 — **애초에 호출로 받을지, DB 를 직접 읽을지**
> — 를 다룬다.

## 무엇을 견주는가

| | 직접 DB 커넥션 | gRPC 조회 |
| --- | --- | --- |
| 화면 추가 비용 | 쿼리만 짜면 됨 — 빠르다 | proto·rpc 추가 필요 |
| 스키마 변경 | 정산 스키마 바뀌면 어드민도 깨짐 | proto 계약만 지키면 내부는 자유 |
| 쿼리 로직 | 어드민이 다시 구현(중복) | 정산의 기존 유스케이스 재사용 |
| 쓰기(상태 변경) | 도메인 규칙을 우회하거나 중복 구현 | 정산이 규칙을 계속 보장 |
| 장애 결합 | 정산이 죽어도 DB 만 살아 있으면 조회 가능 | 정산 서비스가 떠 있어야 함 |
| 보안 | 어드민이 전 서비스 DB 자격증명 보유 | 서비스별 gRPC 엔드포인트만 노출 |

## 직접 DB 커넥션이 나은 경우

- **읽기 전용 리포팅 화면을 빨리 늘려야 한다.** 화면마다 상대 팀에 rpc 를 요청하고
  기다리는 왕복이 없다. 어드민 화면이 수십 개로 늘어나는 큰 조직에서 이 속도 차이는 크다.
- **서비스에 없는 임의 집계가 잦다.** 애드혹 쿼리·크로스 테이블 조인은 API 로 일반화하기
  어렵다. DB 를 직접 읽으면 SQL 한 줄이다.
- **원 서비스가 죽어도 어드민 조회는 살아야 한다.** DB 만 살아 있으면 화면이 뜬다.

## gRPC 조회가 나은 경우

- **서비스별 DB 분리를 지키고 싶다.** 어드민이 각 DB 에 커넥션을 맺는 순간, 스키마가
  사실상 공용 계약이 된다. 정산이 컬럼 하나 바꾸려면 어드민 쿼리까지 봐야 하고, 이는
  DB 를 서비스별로 쪼갠 이유를 되돌리는 것이다.
- **조회 로직이 이미 서비스에 있다.** 정산 목록·요약·상세는 페이징, 표시상태 매핑,
  판매자명 병합(gRPC 배치 조회)까지 `SettlementUseCase` 에 구현돼 있다. 어드민이 DB 를
  직접 읽으면 이걸 어드민 쪽에 다시 짜야 하고, 정산 로직이 바뀔 때마다 두 벌이 어긋난다.
- **쓰기가 섞여 있다.** 상태 전이 규칙(승인·지급 전이 가능 여부)은 정산 도메인 모델의
  불변식이다. 어드민이 DB 를 직접 UPDATE 하면 이 검증을 우회한다. 쓰기는 어차피 정산
  호출이어야 하는데, 그러면 "읽기는 DB, 쓰기는 호출"로 경로가 두 갈래가 된다.

## 정산에 맞는 선택 — 직접 DB 커넥션

어드민은 유저·상품·주문·정산 등 **여러 도메인 데이터를 모아 보여주는 전사 모듈**이다.
화면 하나 추가할 때마다 상대 서비스에 proto·rpc 를 뚫고 합의를 기다리는 왕복을 도메인
수만큼 반복하는 대신, 각 도메인 DB 에 커넥션을 맺어 필요한 걸 직접 읽는다. 정산도 이
구조를 따른다 — **조회(목록·요약·상세·잡 상태)와 쓰기(상태 변경) 모두 어드민이 정산
DB 를 직접 접근**한다.

- **전사 어드민의 접근 방식이 하나로 통일된다.** 도메인마다 어드민용 gRPC 서버를 새로
  여는 대신, 어드민이 각 도메인 DB 를 바라보는 같은 모양이 반복된다. 정산만 gRPC 로
  가르면 어드민 안에서 도메인마다 접근 방식이 갈린다.
- **화면 추가가 빠르다.** 쿼리만 짜면 화면이 나온다. 어드민 화면은 앞으로 도메인 전반으로
  늘어날 예정이라, 이 속도 차이가 누적된다.
- **"쓰기는 어차피 정산 호출" 논거는 예약 테이블이 해소한다.** 배치 실행은 어드민이 정산
  DB 의 **배치 예약 테이블에 INSERT** 하고 정산의 폴링 스케줄러가 집어 실행하는 방식으로,
  쓰기까지 DB 단일 경로에 얹는다. 정산 프로세스를 직접 호출할 일이 없다.
  (구조는 `../architecture/admin-module-separation.md`)

### 감수하는 비용

직접 DB 의 대가는 그대로 남는다. 없는 셈 치지 않고 규칙으로 관리한다.

- **정산 스키마가 어드민과의 사실상 공용 계약이 된다.** 정산 테이블의 컬럼 변경·리네이밍은
  어드민 쿼리를 깨뜨릴 수 있다 — 스키마 변경 시 어드민 영향 확인을 필수 절차로 둔다.
- **조회·상태 전이 로직이 두 벌이 된다.** 페이징, 표시상태 매핑, 판매자명 병합은 정산
  `SettlementUseCase` 에 이미 있지만 어드민이 자기 쿼리로 다시 구현한다. 상태 전이
  가능 여부(승인·지급 전이 규칙)도 어드민 UPDATE 조건으로 재구현된다. 정산 도메인
  모델의 불변식과 어긋나지 않도록 **상태 전이 규칙을 문서 한 곳(단일 출처)에 두고 양쪽이
  따른다.**
- **어드민이 각 서비스 DB 자격증명을 보유한다.** 어드민 전용 DB 계정을 만들어 필요한
  테이블에 최소 권한만 준다.

## 정리

- 어드민의 정산 데이터 접근은 **gRPC 조회가 아니라 직접 DB 커넥션**으로 한다. 조회와
  쓰기(상태 변경) 모두 해당한다.
- 근거: 전사 어드민의 도메인별 접근 방식 통일, 화면 추가 속도, 배치 실행까지 예약 테이블로
  DB 단일 경로 유지.
- gRPC 조회의 장점(스키마 결합 회피·기존 유스케이스 재사용·도메인 규칙의 정산 측 보장)은
  인정하되 비용으로 감수한다. 스키마 변경 시 어드민 영향 확인, 상태 전이 규칙 단일 출처
  문서화, 어드민 전용 최소 권한 DB 계정을 관리 규칙으로 둔다.
- 배치 예약 테이블 구조·폴링 방식은 `../architecture/admin-module-separation.md` 에 정리한다.
