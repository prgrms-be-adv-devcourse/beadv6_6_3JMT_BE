# 음수 정산 이월 (carry-forward)

한 판매자의 월 정산 순액이 음수가 될 수 있다. 이때 어떻게 처리할지를 정한다.
`order-data-sourcing.md` §음수 정산에서 "존재만 명시하고 규칙은 별도로 정한다"고 미뤄둔 그 정책을
여기서 결정한다.

## 문제 — 왜 음수가 나나

정산은 시각(`paidAt`·`refundedAt`) 기준으로 결제·환불 라인을 각각 잡고, 확정된 과거 정산은
소급 수정하지 않는다(회계 원칙). 그래서 다음이 발생한다.

```
6월 결제(이미 6월 정산 확정) + 7월 환불
  → 6월 정산은 안 건드림
  → 7월에 −환불 보정 라인만 추가
```

7월에 그 판매자의 신규 결제액이 6월분 환불액보다 적으면 **7월 순액이 음수**가 된다.
(시각 기준 조회·보정 라인 구조는 `order-data-sourcing.md` 참고.)

## 결정 — 다음 달로 이월한다

음수분을 판매자에게 직접 청구(clawback)하지 않고, **다음 정산 기간으로 넘겨 다음 달 양수 정산에서
차감**한다.

선택지 비교:

| 방식 | 장점 | 단점 |
| --- | --- | --- |
| **이월 (선택)** | 판매자 직접 청구 없음; 지급 흐름은 "이번 달 제외"로 흡수; 커머스 정산의 통상 방식 | 잔액이 다음 달로 넘어가는 상태 관리 필요 |
| 마이너스 확정 + 청구 | 회수 근거가 그 달에 남음 | 청구·회수 프로세스와 별도 채권 관리 필요 — 가장 무거움 |
| 0 클램프 + 미수금 분리 | 지급 로직 단순 | 미수금 상계 시점·규칙을 따로 설계해야 함 |

## 표현 — 음수 Settlement 레코드 + `CARRIED_OVER`

이월 잔액을 **별도 테이블이나 컬럼이 아니라, 그달의 음수 `Settlement` 레코드 자체로** 표현한다.

- 순액 ≥ 0 → 기존대로 `Settlement` 생성. 승인 → 지급 신청 → 지급 흐름을 탄다.
- 순액 < 0 → `Settlement`을 **음수 `totalAmount`로 생성**하되 `payoutStatus = CARRIED_OVER`.
  지급 흐름(승인·신청·지급)에 진입하지 않는다.

```
7월: Settlement(seller=A, totalAmount=-50,000, payoutStatus=CARRIED_OVER)
```

## 소진(folding) — 다음 달 순액에 접어넣고 중복 반영을 막는다

기간 배치가 판매자별 순액을 낼 때, 그 판매자의 **활성 이월 잔액도 함께 읽어** 합산한다.

```
그달 순액 = Σ(PAID) − Σ(REFUND) + 미소진 이월 잔액
```

- 다음 달 순액에 이월분을 반영한 뒤, 흡수한 음수 `Settlement`은 **종료 상태로 소진 처리**한다
  (`CARRIED_OVER → SETTLED`). 이래야 그다음 달에 또 읽히지 않는다.
- 흡수한 `Settlement`에 "어느 이월분을 접어넣었는지" 참조를 남기면 감사 추적이 깔끔하다.
- **판매자당 활성 `CARRIED_OVER`는 항상 최대 1건이다.** 매달 이월분이 그달 순액에 folding 되어
  새 레코드 하나로 다시 표현되기 때문이다. 아래 케이스 A가 그 예다.

### 케이스 (7월 이월 −50,000을 안고 8월 계산)

**케이스 A — 8월 매출이 이월을 다 못 덮음** (8월 그달 순액 +30,000)
```
8월 순액 = +30,000 + (−50,000) = −20,000
8월 지급액 = 0
미차감 잔액 = −20,000        ← 다시 이월. 7월 건은 SETTLED로 소진
```

**케이스 B — 8월 매출이 이월을 넘김** (8월 그달 순액 +70,000)
```
8월 순액 = +70,000 + (−50,000) = +20,000
8월 지급액 = 20,000          ← 여기서 처음으로 실제 지급 발생
미차감 잔액 = 0              ← 이월 종료. 7월 건은 SETTLED로 소진
```

실지급은 **이월 잔액이 소진되고 순액이 양수로 올라서는 순간부터** 발생한다. 그전까지는 0원 지급 +
잔액 이월이 반복된다.

## 미차감 잔액 — 저장하지 않고 계산으로 도출한다

"미차감 잔액"을 DB에 별도 컬럼으로 두지 않는다. 위 표현 구조에서 잔액은 이미 데이터에 들어있다.

```
미차감 잔액 = 그 판매자의 활성 CARRIED_OVER Settlement.totalAmount
```

folding 덕에 활성 `CARRIED_OVER`가 최대 1건이라, SUM도 없이 그 한 건의 `totalAmount`가 곧 미차감
잔액이다. 지급액 0은 `payoutStatus == CARRIED_OVER`에서 도출된다. **순정산액·지급액·미차감 잔액
세 값이 전부 `Settlement` 한 레코드에서 나온다.**

별도 컬럼으로 저장하면 같은 사실을 두 곳(음수 `totalAmount` + 복사한 잔액 컬럼)에 적어 drift 위험이
생기고, `domain-model.md §5`("상세·원본으로 계산 가능한 파생값은 따로 저장하지 않는다")에 어긋난다.
미차감 잔액은 전형적인 파생값이다.

→ 화면 응답의 세 값은 `~Response`의 정적 팩토리(`from`)에서 `Settlement` 하나로 만들어 내려준다.
배치·조회·화면이 전부 같은 레코드를 진실의 원천으로 본다.

## 상태 축 정리

`Settlement`은 승인 축(`settlementStatus`)과 지급 축(`payoutStatus`)을 갖는다. 음수 건은
**승인·지급 흐름을 타지 않고 `payoutStatus = CARRIED_OVER` 하나로만 관리**한다.

- 배치가 음수 건을 만들 때 승인 대기(`PENDING_APPROVAL`)로 두지 않는다. 어드민이 승인할 대상이
  아니라 다음 달에 자동 흡수될 이월분이기 때문이다.
- `CARRIED_OVER`는 지급 축의 종점이 아니라 "다음 배치가 소진할 대기 상태"다. 소진되면 `SETTLED`로
  전이한다. (지급 축 상태·지급 신청 흐름은 `2026-06-24-seller-payout-request-design.md` 참고.)

## 범위에서 뺀 것 (YAGNI)

- **영구 미소진 이월** — 판매자가 이탈해 이후 정산이 잡히지 않으면 음수 잔액이 소진되지 않는다.
  이걸 언제 종결·청구할지는 별도 정책이라 지금 범위에서 뺀다.
- **정산서(명세서) 스냅샷** — 발급 시점 숫자를 영구 불변으로 고정해야 하는 요구가 생기면, 그때
  미차감 잔액을 정산서에 스냅샷으로 박는다. 현재 요구 없음.

## 코드에 닿는 지점

- **배치 계산부** — 판매자별 순액 산출 시 활성 `CARRIED_OVER` 잔액을 읽어 합산. 음수면 음수
  `Settlement` + `CARRIED_OVER` 생성, 흡수한 이월 건은 `SETTLED`로 전이.
- **도메인(`Settlement`)** — `CARRIED_OVER`·`SETTLED` 상태 전이 메서드와 불변식(음수 건은 승인 흐름
  미진입) 추가.
- **조회 응답(`~Response`)** — 정적 팩토리에서 순정산액·지급액·미차감 잔액 도출.
- **그대로** — `settlement_source_line` 적재, 시각 기준 gRPC pull, 지급 신청 흐름.
