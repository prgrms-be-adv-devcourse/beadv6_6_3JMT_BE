# 범용 PR 생성·변경 검증 스킬 설계

## 목표

현재 정산 전용으로 작성된 PR 생성과 규칙 검증 스킬을 저장소 전체 diff에 사용할 수 있도록 범용화한다. 스킬 파일과 관련 문서는 저장소 루트가 아니라 `settlement-service/` 아래에 유지한다.

## 구조

- `settlement-service/.codex/skills/create-project-pr/`: 저장소의 모든 작업 브랜치에서 PR을 준비하고 생성하는 범용 스킬
- `settlement-service/.codex/skills/verify-project-changes/`: 현재 diff 전체의 코드 품질과 규칙을 검증하는 범용 스킬
- `create-settlement-pr`, `verify-settlement-rules`: 기존 호출을 유지하는 호환용 진입점

호환용 스킬은 범용 스킬을 필수로 호출하며 `settlement-service/` 경로로 검증 범위를 다시 제한하지 않는다.

## 전체 diff 검증

1. base 브랜치 대비 commit, 변경 파일, stat, diff와 작업 트리 변경을 수집한다.
2. 변경 파일을 최상위 모듈과 공통 영역으로 분류한다.
3. 각 변경 경로에 적용되는 `AGENTS.md`, `CLAUDE.md`, `.claude/rules/*.md`를 찾아 필요한 원본을 완전히 읽는다.
4. 규칙 파일이 있는 모듈은 해당 규칙을 적용하고, 규칙 파일이 없는 모듈도 검증 대상에서 제외하지 않는다.
5. 전체 diff에서 정확성, 회귀 가능성, 계층과 계약, 예외 처리, 테스트 적정성, 보안, 설정 영향을 검토한다.
6. 변경된 모듈별 테스트·check·build 작업을 선택한다. 공통 모듈이나 루트 설정 변경은 실제 영향 범위까지 검증을 확대한다.
7. 실행하지 못한 검증은 통과로 표시하지 않고 이유와 남은 위험을 기록한다.

정산 규칙은 정산 변경 파일에만 적용한다. payment, product, user 등 다른 모듈에는 각 모듈 규칙을 적용하되, 특정 모듈에 규칙이 없다는 이유로 코드 검토를 생략하지 않는다.

## PR 생성

PR 생성 여부는 변경 파일이 `settlement-service/`에 포함되는지와 무관하다. 범용 변경 검증 결과를 먼저 확인하고, 저장소 PR 템플릿과 기존 기본값을 사용해 초안을 작성한다. 사용자 승인 후 push와 PR 생성 또는 기존 PR 갱신을 수행한다.

기존 assignee, label, reviewer, Ready 기본값과 원격 파일 링크 검증 절차는 유지한다.

## 검증 시나리오

- 정산 파일만 변경된 diff: 정산 규칙과 정산 모듈 테스트를 적용한다.
- 정산과 다른 서비스가 함께 변경된 diff: 모든 변경 파일을 검토하고 각 모듈 규칙과 테스트를 적용한다.
- 정산 파일이 없는 diff: PR 생성을 중단하지 않고 변경된 모듈만 검토한다.
- 규칙 문서가 없는 모듈 변경: 일반 코드 품질·보안·테스트 검토를 수행하고 규칙 부재를 명시한다.
- 루트 또는 공통 모듈 변경: 영향을 받는 하위 모듈까지 검증 범위를 확장한다.

## 범위 밖

- 저장소 루트에 `.codex`, `AGENTS.md` 또는 스킬 라우팅 파일을 새로 만들지 않는다.
- 각 서비스의 기존 규칙 원본을 범용 규칙으로 합치거나 수정하지 않는다.
- PR 생성과 변경 검증 외의 이슈·브랜치·커밋 스킬은 변경하지 않는다.
